## 1. 项目概述

### 1.1 背景与目标
为解决校园二手交易信息不对称、信任成本高的问题，我们构建一个基于 SpringBoot 3 和 Vue 3 的轻量化平台。**核心目标是提供一个功能完整、逻辑严密、代码流畅的校园内部交易市场，为后续 Epic/Story 拆分提供清晰的蓝图。**

### 1.2 核心业务流程
`访客 -> 注册/登录 -> (买家)搜索/浏览 -> 查看商品 -> 咨询 -> 下单 -> (卖家)确认订单 -> 线下交易 -> (买家)确认收货 -> 双方互评 -> 交易完成`

## 2. 详细功能需求

### 2.1 用户管理模块

*   **Epic**: 作为一名用户，我希望能安全地管理我的账户、信息和信誉。

*   **Story 1: 用户注册**
    *   **规则**: 用户名 (4-16位, 字母/数字/下划线, 唯一), 密码 (6-20位, 字母+数字)。密码需 BCrypt 加盐哈希存储。
    *   **流程**: 校验通过 -> 创建用户记录 (初始信用分100) -> 自动登录 -> 返回 JWT。

*   **Story 2: 用户登录与认证**
    *   **规则**: 登录成功后，生成 JWT (Payload含 `userId`, `role`)，有效期7天。客户端需在请求头 `Authorization` 中携带 `Bearer <token>`。

*   **Story 3: 信用评级体系**
    *   **算法**: 初始100分。成功交易+2，好评+1；卖家取消-10，买家取消-5；违规-20。
    *   **等级**: `>150`极好, `120-149`优秀, `100-119`良好, `80-99`一般, `<80`较差。

### 2.2 商品管理模块

*   **Epic**: 作为用户，我希望能方便地发布、管理和发现商品。

*   **Story 1: 商品发布/编辑**
    *   **字段约束**: 标题(1-30字), 描述(1-1000字), 图片(1-9张, 前端压缩<2MB), 价格(>0)。
    *   **规则**: 未完成交易的商品，仅可修改描述/图片。

*   **Story 2: 商品状态机**
    *   `ACTIVE` (在售) -> `LOCKED` (被下单后锁定) -> `SOLD` (交易完成后)。
    *   `LOCKED` -> `ACTIVE` (订单被取消/拒绝)。
    *   `ACTIVE` -> `OFFLINE` (卖家手动下架)。
    *   `*` -> `REMOVED_BY_ADMIN` (管理员强制下架)。

*   **Story 3: 搜索与筛选**
    *   **搜索**: `LIKE '%keyword%'` 匹配标题和描述。
    *   **排序**: 支持按 `create_time` (最新), `price` (升/降), `seller.credit_score` (信用) 排序。

### 2.3 交易管理模块

*   **Epic**: 作为用户，我希望能顺畅地完成交易并与对方沟通。

*   **Story 1: 订单状态机**
    *   `PENDING` (待确认): 买家下单后。卖家可“确认”/“拒绝”，买家可“取消”。
    *   `CONFIRMED` (待交易): 卖家确认后。买家可“确认收货”。
    *   `COMPLETED` (已完成): 买家确认收货后。触发信用分计算。
    *   `CANCELLED` (已取消): 流程中取消。商品状态恢复 `ACTIVE`。

*   **Story 2: 商品咨询/私信**
    *   **实现**: 基于 WebSocket 实现实时消息推送。为每个“买家-卖家”的首次咨询创建唯一会话。

### 2.4 管理后台模块

*   **Epic**: 作为管理员，我需要高效地监控和管理平台。

*   **Story 1: 仪表盘**
    *   **指标**: 总用户/今日新增，总商品/今日上架，总成交额/今日成交额。
    *   **图表**: 近7日用户增长折线图，商品分类饼图。

*   **Story 2: 用户/商品管理**
    *   **操作**: 支持对用户进行“封禁”/“解封”，对商品进行“强制下架”。操作需记录日志并通知用户。

## 3. 技术实现概要

### 3.1 API 端点设计 (RESTful)
*   **Auth**: `POST /api/auth/register`, `POST /api/auth/login`
*   **Items**: `GET /api/items`, `GET /api/items/{id}`, `POST /api/items`
*   **Orders**: `POST /api/orders`, `GET /api/orders/my`, `PUT /api/orders/{id}/confirm`
*   **Admin**: `GET /api/admin/stats`, `GET /api/admin/users`, `PUT /api/admin/users/{id}/ban`

### 3.2 数据字典
| 表名 | 字段名 | 数据类型 | 约束/注释 |
| :--- | :--- | :--- | :--- |
| **users** | `id`, `username`, `password`, `credit_score`, `status` | `BIGINT`, `VARCHAR(50)`, `VARCHAR(255)`, `INT`, `TINYINT` | PK, UNIQUE, NOT NULL, DEFAULT 100, 0:正常/1:封禁 |
| **items** | `id`, `seller_id`, `title`, `price`, `status` | `BIGINT`, `BIGINT`, `VARCHAR(100)`, `DECIMAL(10,2)`, `TINYINT` | PK, FK, NOT NULL, >0, 0:在售/1:锁定/2:已售出 |
| **orders** | `id`, `item_id`, `buyer_id`, `status` | `BIGINT`, `BIGINT`, `BIGINT`, `TINYINT` | PK, FK, FK, 0:待确认/1:待交易/2:已完成/3:已取消 |
| **item_images**| `id`, `item_id`, `image_url` | `BIGINT`, `BIGINT`, `VARCHAR(255)` | PK, FK, NOT NULL |

## 4. 验收标准
1.  **端到端流程通畅**: 可无阻碍地完成“注册-发布-下单-交易-评价”的完整闭环。
2.  **状态机准确**: 商品和订单状态的流转必须与业务规则完全一致。
3.  **数据一致性**: 信用分、订单金额、商品库存等核心数据的计算和更新必须准确无误。
4.  **后台管理有效**: 管理员的封禁、下架等操作能实时生效并产生正确影响。
